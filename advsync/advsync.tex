虽然无锁算法在面临极端需求时可以提供很大帮助，但它们绝不是万能的。

内存排序也非常重要，但它值得拥有自己的

高级同步的第二种形式提供了并行real-time计算所需的更强的

\section{避免锁}
\label{sec:advsync:Avoiding Locks}
%
\epigraph{我们面临着不可逾越的机遇。}
	 {Walt Kelly}

虽然锁是生产环境中并行性的主力，但在许多情况下，
通过使用无锁技术可以大大改善性能、可扩展性和real-time响应。
这种无锁技术的一个特别令人印象深刻的例子是
\cref{sec:count:Statistical Counters}中描述的统计计数器（statistical counter），
它不仅避免了锁，还避免了读-修改-写原子操作（atomic operation）、
内存屏障（memory barrier），甚至避免了计数器递增时的缓存未命中（cache miss）。
我们已经涵盖的其他例子包括：

\begin{enumerate}
\item	\cref{chp:Counting}中若干其他计数算法的快速路径。
\item	\cref{sec:SMPdesign:Resource Allocator Caches}中
	资源分配器缓存的快速路径。
\item	\cref{sec:SMPdesign:Beyond Partitioning}中的迷宫求解器。
\item	\cref{chp:Data Ownership}中的数据所有权技术。
\item	\cref{chp:Deferred Processing}中的引用计数（reference counting）、hazard pointer和RCU技术。
\item	\cref{chp:Data Structures}中的查找代码路径。
\item	\cref{chp:Putting It All Together}中的许多技术。
\end{enumerate}

简而言之，无锁技术非常有用且被大量使用。
然而，最好是将无锁技术隐藏在定义良好的API之后，
例如\co{inc_count()}、\co{memblock_alloc()}、
\co{rcu_read_lock()}等。
这样做的原因是，无纪律地使用无锁技术是制造
难以发现的bug的好方法。
如果你相信发现和修复这类bug比避免它们更容易，
请重新阅读
\cref{chp:Validation,chp:Formal Verification}。

\section{非阻塞同步}
\label{sec:advsync:Non-Blocking Synchronization}
%
\epigraph{只要机器按预期工作，就永远不用担心理论。}
	 {Robert A. Heinlein}

术语\IXBacrfst{nbs}~\cite{MauriceHerlihy90a}
描述了具有不同\emph{\IXBpl{forward-progress guarantee}}的
八类\IX{linearizable}算法~\cite{DanAlitarh2013PracticalProgress}，
具体如下：

\begin{enumerate}
\item	\emph{\IXalth{Bounded population-oblivious wait-free}{bounded population-oblivious}{wait free}同步}：
	每个线程（thread）都将在特定的有限时间内取得进展，
	且该时间段与线程数量无关~\cite{HerlihyShavit2008Textbook}。
	人们普遍认为这个层次甚至比bounded wait-free同步更难实现。
\item	\emph{\IXalth{Bounded wait-free}{bounded}{wait free}同步}：
	每个线程都将在
	特定的有限时间内取得进展~\cite{Herlihy91}。
	人们普遍认为这个层次是不可实现的，这也许是
	Alitarh等人将其省略的原因~\cite{DanAlitarh2013PracticalProgress}。
\item	\emph{\IXalt{Wait-free}{wait free}同步}：
	每个线程都将在
	有限时间内取得进展~\cite{Herlihy93}。
\item	\emph{\IXalt{Lock-free}{lock free}同步}：
	至少有一个线程将在
	有限时间内取得进展~\cite{Herlihy93}。
\item	\emph{\IXalt{Obstruction-free}{obstruction free}同步}：
	在没有竞争的情况下，每个线程都将在有限时间内
	取得进展~\cite{HerlihyLM03}。
\item	\emph{\IXalt{Clash-free}{clash free}同步}：
	在没有竞争的情况下，至少有一个线程将在有限时间内
	取得进展~\cite{DanAlitarh2013PracticalProgress}。
\item	\emph{\IXalt{Starvation-free}{starvation free}同步}：
	在没有故障的情况下，每个线程都将在有限时间内
	取得进展~\cite{DanAlitarh2013PracticalProgress}。
\item	\emph{\IXalt{Deadlock-free}{deadlock free}同步}：
	在没有故障的情况下，至少有一个线程将在有限时间内
	取得进展~\cite{DanAlitarh2013PracticalProgress}。
\end{enumerate}

NBS第1类是在2015年之前的某个时期提出的，
第2、3、4类最初是在1990年代初期提出的，
第5类最初是在2000年代初期提出的，
第6类最初是在2013年提出的。
最后两类已被非正式使用了很多年，
但在2013年被重新表述。

\QuickQuiz{
	考虑到可用的CPU数量总是有严格限制的，
	population obliviousness真的有用吗？
}\QuickQuizAnswer{
	考虑到许多NBS算法令人惊讶的有限可扩展性，
	population obliviousness可以出乎意料地有用。
	尽管如此，这个问题的总体观点是正确的。
	一个算法的扩展能力超过它将要运行的最大系统的规模，
	通常是没有帮助的。
}\QuickQuizEnd

理论上，任何并行算法都可以转换为wait-free形式，
但在实际中常用的NBS算法只是相对较小的一个子集。
下一节列出了其中的一些。

\subsection{简单NBS}
\label{sec:advsync:Simple NBS}

也许最简单的NBS算法是使用fetch-and-add
（\co{atomic_add_return()}）原语对整数计数器进行原子更新。
本节按大致递增的复杂度列出了一些其他常用的NBS算法。

\subsubsection{NBS集合}
\label{sec:advsync:NBS Sets}

一种简单的NBS算法使用数组实现整数集合。
这里数组索引表示可能属于集合成员的值，
数组元素表示该值是否确实是集合成员。
NBS算法的linearizability准则要求对数组的读取和更新
要么使用原子指令，要么配合内存屏障，
但在linearizability不重要的常见情况下，
简单的volatile加载和存储就足够了，
例如使用\co{READ_ONCE()}和\co{WRITE_ONCE()}。

NBS集合也可以使用位图来实现，其中每个可能属于集合成员的值
对应一个位。
读取和更新通常必须通过原子位操作指令来进行，
尽管compare-and-swap（\co{cmpxchg()}或CAS）
指令也可以使用。

\subsubsection{NBS计数器}
\label{sec:advsync:NBS Counters}

\cref{sec:count:Statistical Counters}中讨论的统计计数器算法
可以被认为是bounded-wait-free的，但这只是通过一个巧妙的
定义技巧实现的，即将求和视为近似值
而非精确值。\footnote{
	需要引用来源。
	我从Mark Moir口中听到了这个技巧。}
给定足够宽的误差范围（该范围是\co{read_count()}函数
对计数器求和所需时间的函数），
就不可能证明任何非linearizable行为发生了。
这确实（虽然有些人为地）将统计计数器算法
归类为bounded wait-free。
这个算法可能是Linux内核中使用最广泛的NBS算法。

\subsubsection{半NBS队列}
\label{sec:advsync:Half-NBS Queue}

\begin{fcvref}[ln:advsync:NBS Enqueue Algorithm]
另一种常见的NBS算法是原子队列，其中元素使用原子交换指令
入队~\cite{MagedMichael1993JPDC}，
然后存储到新元素的前驱节点的\co{->next}指针中，
如\cref{lst:advsync:NBS Enqueue Algorithm}所示，
该图展示了用户空间RCU库的
实现~\cite{MathieuDesnoyers2009URCU}。
\Clnref{tail}更新尾指针以引用新元素，同时返回
对其前驱节点的引用，该引用存储在
局部变量\co{old_tail}中。
\Clnref{pred}然后更新前驱节点的\co{->next}指针以
引用新添加的元素，最后\clnref{ret}
返回队列最初是否为空的指示。

\begin{listing}

\label{lst:advsync:NBS Enqueue Algorithm}
\end{listing}

虽然出队单个元素需要互斥（因此出队是阻塞的），
但可以对队列的全部内容执行非阻塞移除。
不可能做到的是以非阻塞方式出队任何给定元素：
入队者可能在该清单的\clnref{tail,pred}之间失败，
因此相关元素只是部分入队。
这导致了一种半NBS算法，其中入队是NBS的，
但出队是阻塞的。
尽管如此，这个算法在实践中被大量使用，部分原因是
大多数生产软件不需要容忍任意的fail-stop错误。
\end{fcvref}

\QuickQuiz{
	等一下！
	为了出队所有元素，必须同时更改\co{->head}和
	\co{->tail}指针，而这在典型的计算机系统上
	无法原子地完成。
	那么这应该怎么工作呢？？？
}\QuickQuizAnswer{
	一次一个指针！

	首先，原子地将\co{->head}指针与\co{NULL}交换。
	如果原子交换操作的返回值是\co{NULL}，
	则队列为空，操作完成。
	如果此时其他人尝试出队全部，
	他们将得到一个\co{NULL}指针。

	否则，原子地将\co{->tail}指针与指向
	现在为\co{NULL}的\co{->head}指针的指针交换。
	原子交换操作的返回值是指向列表最终最后一个
	元素的\co{->next}字段的指针。

	编写和测试实际代码留给感兴趣且有热情的读者作为练习，
	处理半入队元素的策略也是如此。
}\QuickQuizEnd

\subsubsection{NBS栈}
\label{sec:advsync:NBS Stack}

\begin{fcvref}[ln:advsync:lifo_push:whole]
\Cref{lst:advsync:NBS Stack Algorithm}
展示了LIFO push算法，该算法具有lock-free的push和
bounded wait-free的pop（\path{lifo-push.c}），形成了一个NBS栈。
这个算法的起源未知，但在1975年授予的一项专利中
被引用过~\cite{PaulJBrown1975LIFOpush}。
该专利于1973年提交，比本书编辑第一次见到计算机
早几个月，而那台计算机只有一个CPU\@。

\begin{listing}

\end{listing}

\Clnrefrange{struct:b}{struct:e}展示了\co{node_t}结构，
其中包含一个任意值和一个指向栈中下一个结构的指针，
\clnref{top}展示了栈顶指针。

\co{list_push()}函数跨越\clnrefrange{push:b}{push:e}。
\Clnref{push:alloc}分配一个新节点，
\clnref{push:initialize}对其进行初始化。
\Clnref{push:next}初始化新分配节点的\co{->next}
指针，\clnref{push:cmpxchg}尝试将其压入栈中。
如果\clnref{push:check}检测到\co{cmpxchg()}失败，
则循环的另一次遍历重试。
否则，新节点已成功压入，函数返回给调用者。
注意\clnref{push:check}解决了两个并发
\co{list_push()}实例试图压入栈时的竞争。
\co{cmpxchg()}对其中一个成功，对另一个失败，
导致另一个重试，从而为栈上的两个节点选择任意顺序。

\co{list_pop_all()}函数跨越\clnrefrange{popall:b}{popall:e}。
\clnref{popall:xchg}上的\co{xchg()}语句原子地移除
栈上的所有节点，将结果列表的头部放入局部
变量\co{p}中，并将\co{top}设置为\co{NULL}。
这个原子操作序列化了对\co{list_pop_all()}的并发调用：
其中一个将获得列表，另一个获得\co{NULL}指针，
至少假设没有对\co{list_push()}的并发调用。

获得非空列表的\co{list_pop_all()}实例
在跨越\clnrefrange{popall:loop:b}{popall:loop:e}的循环中
处理此列表。
\Clnref{popall:next}预取\co{->next}指针，
\clnref{popall:foo}对当前节点调用\co{foo()}引用的函数，
\clnref{popall:free}释放当前节点，
\clnref{popall:pnext}为循环的下一次遍历设置\co{p}。

但假设一对\co{list_push()}实例与一个\co{list_pop_all()}
并发运行，且列表最初包含一个单独的\Node{A}。
以下是这种场景可能发生的一种方式：

\begin{enumerate}
\item	第一个\co{list_push()}实例压入一个新的\Node{B}，
	执行到\clnref{push:next}，刚刚将
	指向\Node{A}的指针存储到\Node{B}的\co{->next}指针中。
\item	\co{list_pop_all()}实例运行完成，
	将\co{top}设置为\co{NULL}并释放\Node{A}。
\item	第二个\co{list_push()}实例运行完成，
	压入一个新的\Node{C}，但恰好分配了
	原来属于\Node{A}的内存。
\item	第一个\co{list_push()}实例执行
	\clnref{push:cmpxchg}上的\co{cmpxchg()}。
	因为新\Node{C}与新释放的\Node{A}具有相同的地址，
	此\co{cmpxchg()}成功，且此\co{list_push()}实例
	运行完成。
\end{enumerate}

注意，尽管\Node{A}的内存被重新使用，
两次push和一次popall都成功运行了。
这是一个不寻常的属性：
大多数数据结构需要对通常称为ABA问题的情况进行保护。

但这个属性仅适用于用汇编语言编写的算法。
令人遗憾的事实是，大多数语言（包括C和C++）不支持
指向生存期已结束对象的指针，例如\Node{B}的
\co{->next}指针中包含的指向旧\Node{A}的指针。
事实上，编译器有权假设如果两个指针（称之为\co{p}和\co{q}）
是从两次不同的\co{malloc()}调用返回的，
那么这些指针必定不相等。
真正的编译器确实会在响应\co{p==q}比较时
生成常量\co{false}。
指向已释放但其内存已被重新分配给兼容类型对象的指针
被称为\emph{僵尸指针}。

许多并发应用程序通过精心对编译器隐藏内存分配器
来避免这个问题，从而防止编译器做出不恰当的假设。
这种混淆方法目前在实践中有效，但很可能有一天
会成为日益激进的优化器的牺牲品。
C和C++标准委员会中都有正在进行的工作来解决这个
问题~\cite{PaulEMcKenney2019PointerLifetimeEndZap,PaulEMcKenney2020PointerLifetimeEndZapCpp}。
在此期间，请在编写ABA容忍算法时格外小心。
\end{fcvref}

\subsection{NBS优势的适用性}
\label{sec:advsync:Applicability of NBS Benefits}

NBS最被广泛引用的优势来源于其前进保证、
对fail-stop错误的容忍以及其linearizability。
以下各节分别讨论其中每一项。

\subsubsection{NBS前进保证}
\label{sec:advsync:NBS Forward Progress Guarantees}

NBS的前进保证使许多人建议将其用于real-time系统，
NBS算法事实上也在大量此类系统中使用。
然而，重要的是要注意，前进保证与构成real-time编程基础的
保证在很大程度上是正交的：

\begin{enumerate}
\item	Real-time前进保证通常有某个确定的时间与之关联，
	例如，``\IXh{scheduling}{latency}必须小于100微秒。''
	相比之下，最流行的NBS形式只保证
	将在有限时间内取得进展，没有确定的界限。
\item	Real-time前进保证通常是概率性的，
	如soft-real-time保证
	``至少99.9\pct{}的时间，调度延迟必须
	小于100微秒。''
	相比之下，许多NBS的前进保证是
	无条件的，这可能带来额外的不必要复杂性
	和性能下降。
\item	Real-time前进保证通常以环境约束为条件，
	例如，仅在以下情况下才被尊重：
	\begin{enumerate*}[(1)]
	\item 对于最高优先级的任务，
	\item 当每个CPU至少将一定比例的时间花在空闲上，
	以及
	\item 当I/O速率低于某个指定的最大值时。
	\end{enumerate*}
	相比之下，NBS的前进保证通常是
	无条件的，再次导致不必要的复杂性和
	性能下降。
	另一方面，一些近期的NBS工作适应了某些
	有条件的
	保证~\cite{DanAlitarh2013PracticalProgress}。
\item	Real-time程序环境的一个重要组成部分
	是调度器。
	NBS算法假设最坏情况下的\emph{恶魔调度器}，
	尽管出于某种原因，不会假设调度器恶魔到
	完全拒绝运行包含NBS算法的应用程序。
	相比之下，real-time系统假设调度器
	正在尽最大努力满足它所知道的任何调度约束，
	并且在没有此类约束的情况下，
	尽最大努力尊重进程优先级并为
	相同优先级的进程提供公平调度。
	非恶魔调度器允许real-time程序使用比
	NBS所需的更简单的
	算法~\cite{DanAlitarh2013PracticalProgress,BjoernBrandenburgPhD}。
\item	Non-blocking synchronization提供的前进保证
	在过载条件下最为有用。
	然而，过载条件最有可能导致排队
	延迟，在许多实际系统中，这会导致程序
	（甚至运行该程序的系统）被强制终止。
	作为一个粗略的经验法则，延迟随空闲时间的
	倒数增长。
	换句话说，如果一个几乎空闲的系统有
	延迟$T$，那么50\pct{}空闲的系统将有延迟$2 T$，
	10\pct{}空闲（90\pct{}利用率）的系统将有延迟$10 T$，1\pct{}
	空闲（99\pct{}利用率）的系统将有延迟$100 T$，依此类推。
	这种情况意味着许多延迟敏感的系统会
	主动限制负载，例如限制到50\pct{}。
\item	在给定的计算结果只是锦上添花而非至关重要的
	常见情况下，使用超时可以使阻塞操作具有
	在实践中通常足够的非阻塞属性。
\item	NBS前进保证类别假设许多底层操作（如系统调用）
	是lock-free甚至wait-free的，
	但事实上这些操作在常见的计算机系统上是阻塞的。
\item	NBS前进保证通常通过细分操作来实现。
	例如，为了避免阻塞的出队操作，
	NBS算法可能会用非阻塞的轮询操作来替代。
	这在理论上是可以的，但对于需要元素及时通过队列
	传播的实际程序来说并没有帮助。
\item	Real-time前进保证通常仅在
	没有软件bug的情况下适用。
	相比之下，许多类别的NBS保证即使在
	面对fail-stop错误时也适用，尽管显然不适用于
	一般性的任意错误行为的软件bug。
\item	NBS前进保证类别隐含着linearizability。
	相比之下，real-time前进保证通常不需要
	linearizability这样的排序约束。
	同样，在不需要linearizability时提供它
	可能会不必要地增加复杂性并降低性能。
\item	虽然更简单的NBS算法兼具典范的简洁性、
	前进保证和性能，但其他NBS算法相当复杂。
	这种增加的复杂性可能导致开发和维护成本增加、
	性能降低和可靠性下降。
\end{enumerate}

\subsubsection{NBS底层操作}
\label{sec:advsync:NBS Underlying Operations}

一个NBS算法只有在其使用的底层操作也是非阻塞的情况下
才能真正是非阻塞的。
在数量惊人的案例中，实践中并非如此。

例如，非阻塞算法通常分配内存。
理论上，这没问题，因为存在lock-free的
内存分配器~\cite{MagedMichael04NBSmalloc}。
但在实践中，大多数环境最终必须从操作系统内核获取内存，
而内核通常使用锁。
因此，除非以某种方式预先分配了所有将来需要的内存，
否则分配内存的"非阻塞"算法在常见的实际计算机系统上
运行时将不是非阻塞的。

同样的观点显然也适用于执行I/O操作或以其他方式
与环境交互的算法。

也许令人惊讶的是，这一点也适用于表面上只进行
普通加载和存储的非阻塞算法，例如
\cref{sec:advsync:NBS Counters}中讨论的计数器。
乍一看，那些可以分别编译为单个加载和存储指令的
加载和存储，似乎不仅是非阻塞的，
而且是bounded population-oblivious wait free的。

然而，加载和存储指令不一定是快速的或确定性的。
例如，正如\cref{chp:Hardware and its Habits}中所指出的，
缓存未命中可能消耗数千个CPU周期。
更糟糕的是，测量到的缓存未命中延迟可能是
CPU数量的函数，如
\cref{fig:count:Atomic Increment Scalability on x86}所示。
可以合理地假设这些延迟还取决于系统互连的细节。
此外，鉴于硬件供应商通常不公布缓存未命中延迟的上界，
假设内存引用指令在现代计算机系统中实际上是wait-free的
似乎是勇敢的。
而那些可获得此类上界的古老系统则存在
深刻的整体慢速问题。

此外，硬件不是内存引用指令慢速的唯一来源。
例如，在典型的计算机系统上运行时，
加载和存储都可能导致页面错误。
这会导致内核中的页面错误处理程序被调用。
处理程序可能获取锁，甚至执行I/O，
可能甚至使用网络文件系统（NFS）之类的东西\@。
所有这些都是明确的阻塞操作。

页面错误也不是唯一的内核引起的危害。
给定的CPU可能随时被中断，
而中断处理程序可能运行一段时间。
在此期间，用户态的名义上非阻塞算法
将完全不运行。
这种情况引发了关于依赖中断的系统调用
所提供的前进保证的有趣问题，例如
\co{membarrier()}系统调用。

情况看起来确实很黯淡，但可以通过多种方法
至少部分恢复此类算法的非阻塞性质：

\begin{enumerate}
\item	在裸机上运行，禁用分页。
	如果你既勇敢又有信心能编写没有野指针bug的代码，
	这种方法可能适合你。
\item	在非阻塞操作系统内核上运行~\cite{Cheriton96a}。
	此类内核非常罕见，部分原因是它们传统上
	完全未能提供期望的性能和可扩展性优势
	相对于基于锁的内核。
	但也许你应该写一个。
\item	使用\co{mlockall()}等设施来避免页面错误，
	同时确保你的程序在启动时预先分配
	所有将来需要的内存。
	这可以工作得很好，但代价是内存在常见情况下
	严重未被充分利用。
	在成本受限或功耗有限的环境中，
	这种方法不太可能是可行的。
\item	使用Linux内核的
	\co{NO_HZ_FULL}无滴答模式等设施~\cite{JonCorbet2013NO-HZ-FULL}。
	在最近版本的Linux内核中，该模式将中断
	引导离指定的CPU集合。
	然而，对于在其运行的某个阶段是I/O密集型的应用程序来说，
	这可能会严重限制吞吐量。
\end{enumerate}

鉴于这些考虑，non-blocking synchronization
在理论上远比在实践中重要也就不足为奇了。

\subsubsection{NBS细分操作}
\label{sec:advsync:NBS Subdivided Operations}

一种常见的技巧是通过将阻塞操作替换为轮询API来使
给定算法在NBS排名中获得更高的位置\@。
例如，不是提供一个可能仅仅是lock-free甚至是阻塞的
可靠出队操作，而是提供一个会以wait-free方式
虚假失败的出队操作，而不是展示
可怕的lock-free或阻塞行为。

这在理论上可以很好地工作，但在实践中的一个常见效果是
仅仅将lock-free或阻塞行为从特定算法中
移到使用该算法的不幸代码中。
在这种情况下，这个技巧不仅没有带来任何收益，
而且增加了该算法所有用户的复杂性。

与并发算法中的其他情况一样，最大化特定指标
不能替代对用户需求的仔细思考。

\subsubsection{NBS Fail-Stop容忍}
\label{sec:advsync:NBS Fail-Stop Tolerance}

在NBS算法的各类中，wait-free同步（不论是否有界）、
lock-free同步、obstruction-free同步
和clash-free同步即使在存在fail-stop错误的情况下
也能保证前进。
一个fail-stop错误的例子可能导致某个线程被无限期地抢占。
正如我们将看到的，这种fail-stop容忍属性可以是有用的，
但事实是，组合一组fail-stop容忍机制
不一定会产生一个fail-stop容忍系统。
为了说明这一点，考虑一个由一系列wait-free队列组成的系统，
其中一个元素从系列中的一个队列中移除、处理，
然后添加到下一个队列。

如果一个线程在队列操作过程中被抢占，理论上
一切都好，因为队列的wait-free性质将保证
前进。
但在实践中，正在处理的元素会丢失，因为
wait-free队列的fail-stop容忍性质不会扩展到
使用这些队列的代码。

尽管如此，仍有少数应用程序中NBS相当有限的
fail-stop容忍性是有用的。
例如，在某些基于网络或Web的应用程序中，
fail-stop事件最终会导致重传，
这将重新启动因fail-stop事件而丢失的任何工作。
因此，运行此类应用程序的系统可以承受很重的负载，
甚至到调度器不再能提供任何合理的
公平性保证的程度。
相比之下，如果一个线程在持有锁时fail-stop，
应用程序可能需要重新启动。
尽管如此，即使在这个受限的领域内，NBS也不是万能的，
因为纯调度延迟可能导致虚假重传。
在某些情况下，降低负载以避免排队延迟
可能更有效，这也将改善调度器提供公平访问的能力，
减少甚至消除fail-stop事件，
从而减少重试操作的数量，进而进一步降低负载。

\subsubsection{NBS 线性一致性}
\label{sec:advsync:NBS Linearizability}

重要的是要注意，linearizability可以非常有用，
特别是在分析由严格锁定和完全有序原子操作
组成的并发代码时。\footnote{
	例如，Linux内核的返回值原子操作。}
此外，这种对完全有序原子操作的处理
自动涵盖了简单的NBS算法。

然而，复杂NBS算法的线性化点通常
深埋在算法内部，因此对实现此类算法一部分的
库函数的用户来说不可见。
因此，任何声称用户从复杂NBS算法的linearizability属性
中受益的说法都应该被深度
怀疑~\cite{AndreasHaas2012FIFOisnt}。

有时有人断言linearizability是开发者
为其并发代码生成正确性证明所必需的。
然而，此类证明是例外而非常规，现代开发者
如果确实生成证明，通常使用不依赖于
linearizability的现代证明技术。
此外，开发者经常使用不需要完整规约的
现代证明技术，因为开发者通常是在事后才了解
其规约的，每次一个bug。
\cref{chp:Formal Verification}中讨论了
一些这样的证明技术。\footnote{
	与一位linearizability的倡导者进行的一次令人难忘的口头讨论
	产生了这个问题：
	``所以linearizability重要的原因是为了
	挽救1980年代的证明技术？''
	该倡导者立即肯定地回答，然后花了一些时间
	贬低某种特定的现代证明技术。
	奇怪的是，那种技术正是成功应用于
	Linux内核RCU的技术之一\@。}

经常有人断言linearizability很好地映射到顺序规约，
据称这比并发规约更
自然~\cite{SergioRajsbaum2020HistoryLinearizability}。
但这种断言未能考虑到我们高度并发的客观世界。
这个世界只能被期望选择应对并发的能力，
特别是对于参加团队运动或
照看小孩子的人来说。
此外，鉴于顺序计算的教学
仍被认为是某种黑魔法~\cite{ElizabethPatitsas2020GradesNotBimodal}，
合理地预期并发计算的教学处于类似的
混乱状态。
因此，只关注一种证明技术不太可能是
一条好的前进道路。

再次强调，请理解linearizability在许多情况下
非常有用。
话虽如此，那个古老的工具——锤子——也是如此。
但在计算领域总有一个时刻，人们应该放下
锤子，拿起键盘。
同样，似乎在某些时候linearizability不是
最适合工作的工具。

值得赞扬的是，一些linearizability的倡导者意识到了
它的一些缺点~\cite{SergioRajsbaum2020HistoryLinearizability}。
也有扩展linearizability的提议，例如，
interval-linearizability，旨在处理操作需要
非零时间才能完成的常见
情况~\cite{10.1145/3266457}。
这些提议是否会产生能够处理现代并发软件制品的理论
仍有待观察，特别是考虑到\cref{chp:Formal
Verification}中讨论的几种证明技术已经能处理
许多现代并发软件制品。

\subsection{NBS讨论}
\label{sec:advsync:NBS Discussion}

创建完全非阻塞的队列是可能的~\cite{MichaelScott96}，
然而，这种队列比上面概述的半NBS算法复杂得多。
这里的教训是仔细考虑你的实际需求。
放宽不相关的需求通常可以带来
简洁性、性能和可扩展性的巨大改进。

最近的研究指出了另一种放宽需求的重要方式。
事实证明，提供公平调度的系统可以享受
wait-free同步的大部分好处，即使运行
仅提供non-blocking同步的算法，
无论是在理论上~\cite{DanAlitarh2013PracticalProgress}
还是在实践中~\cite{SamyAlBahra2013NBS}。
因为大多数生产中使用的调度器确实提供公平性，
提供wait-free同步的更复杂算法通常
比更简单、更快的非wait-free算法没有实际优势。

有趣的是，公平调度只是实践中经常被遵守的
一个有益约束。
其他约束集合可以允许阻塞算法
实现确定性的real-time响应。
例如，给定：
\begin{enumerate*}[(1)]
\item 在给定优先级内按FIFO顺序授予的公平锁，
\item priority inversion避免（例如，priority
inheritance~\cite{Takada:1995:RSN:527074.828566,Cai-DongWang1996PrioInherLock}
或priority ceiling），
\item 有界的线程数，
\item 有界的临界区（critical section）持续时间，
\item 有界的负载，
以及
\item 没有fail-stop错误，
\end{enumerate*}
基于锁的应用程序可以提供确定性的
响应时间~\cite{BjoernBrandenburgPhD,DipankarSarma2004OLSscalability}。
这种方法当然模糊了阻塞和wait-free同步之间的区别，
这完全是好事。
希望理论框架能继续改进其描述
实际使用的软件的能力。

那些认为理论应该引领道路的人可以参考
无与伦比的Peter Denning，他在谈论操作系统时说：
``理论跟随实践''~\cite{Denning:2015:POF:2830903.2830904}，
或者杰出的Tony Hoare，他在谈论整个工程学时说：
``在所有工程科学分支中，工程先于科学开始；
事实上，没有早期的工程产品，
科学家就没有什么可以
研究的！''~\cite{RichardMorris2007TonyHoareInterview}。
当然，一旦适当的理论体系可用，
明智地利用它是明智的。
然而，请注意，第一个\emph{适当的}理论体系
往往是一回事，而第一个\emph{被提出的}理论体系
则往往是另一回事。
