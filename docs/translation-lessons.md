# 翻译与质量提升经验教训

## 项目背景

本项目是 Paul McKenney 的《Is Parallel Programming Hard, And, If So, What Can You Do About It?》的中文翻译。在翻译和质量提升过程中，我们积累了以下经验教训。

---

## 一、翻译规范

### 1. 术语一致性

**原则：** 技术术语在整个翻译项目中必须保持一致。

**经验：**
- 建立术语表（terms.yaml）并严格执行
- 术语首次出现时加括号标注英文：`锁（lock）`
- 后续使用统一的中文翻译
- 部分技术术语保留英文：RCU, grace period, CPU, per-CPU

**教训：**
- ❌ "lock" 和 "锁" 混用会严重影响阅读体验
- ❌ "deadlock" 和 "死锁" 混用会造成困惑
- ✅ 统一使用"锁"和"死锁"提高一致性

### 2. 信雅达标准

**信（Faithfulness）：** 忠实原文含义
- 不误译、不遗漏、不添加
- 保持原文语气（幽默、反讽等）
- 准确传达技术概念

**雅（Elegance）：** 优雅的中文表达
- 避免翻译腔
- 符合中文技术写作规范
- 使用地道的中文表达

**达（Expressiveness）：** 清晰流畅传达
- 逻辑清晰
- 句式自然
- 易于理解

## 二、常见翻译问题

### 1. 翻译腔问题

| 问题类型 | 示例 | 修复方法 |
|---------|------|---------|
| 被动语态滥用 | "被...所持有的lock" | 改为主动语态 |
| 的字堆叠 | "通信相对于处理的高overhead" | 重构句式 |
| 过长定语 | "面向那些希望避免学习...以便更好地继续其并发编程生活的人" | 拆分为短句 |
| 英文句式直译 | "这一巨大变化的产生基于两个基本原因" | 改为"源于两个基本原因" |

### 2. 术语问题

| 问题 | 示例 | 修复 |
|------|------|------|
| 术语不一致 | "lock" 和 "锁" 混用 | 统一为"锁" |
| 术语不当 | "惯用法" | 改为"编程惯用法" |
| 术语缺失 | "hazard pointer" 未翻译 | 翻译为"风险指针" |

### 3. 误译问题

| 问题 | 示例 | 修复 |
|------|------|------|
| 词义误解 | "一来" (first time) | 改为"第一次" |
| 概念错误 | "闭源" (in secret) | 改为"保密状态下开发" |
| 信息遗漏 | 缺少重要句子 | 补充翻译 |

## 三、技术实现

### 1. 术语检查工具

**工具：** `check_terms.py`

**改进历程：**
- 初版：2448 个误报
- 改进后：264 个真实问题（减少 89%）

**关键改进：**
1. 首次出现标注检测：排除 "锁（lock）" 等标注
2. 混合使用模式：排除 "读写lock" 等中文+英文模式
3. 词边界匹配：避免 "deadlock" 中的 "lock" 被误报
4. LaTeX 命令排除：`\co{lock}`、`\IX{lock}` 等不再触发
5. 代码环境追踪：Verbatim 环境内的内容不再检查

### 2. 翻译进度统计

**工具：** `check_progress.py`

**功能：**
- 扫描所有 .tex/.md 文件
- 统计可翻译行数和已翻译行数
- 生成进度报告

### 3. CI/CD 集成

**工作流：** GitHub Actions

**设计：**
- 质量检查 job：术语检查 + 进度统计
- 构建 job：生成 PDF
- 上传 job：保存构建产物

## 四、工作流程

### 1. 日常翻译流程

```
1. 检查术语一致性
   python .translate-engine/scripts/check_terms.py

2. 查看翻译进度
   python .translate-engine/scripts/check_progress.py

3. 翻译（人工 + AI 辅助）
   - 使用 Claude Code 翻译
   - 术语自动检查
   - 质量实时评估

4. 构建验证
   make 1c

5. 提交
   git commit
```

### 2. 上游更新流程

```
1. 同步原文
   git subtree pull --prefix=source upstream master

2. 检测变更
   git diff HEAD~1 -- source/

3. 合并 + 翻译变更部分
   git merge upstream/master
   # 解决冲突，翻译新增内容

4. 验证
   make 1c
```

### 3. 质量检查流程

```
1. 术语一致性检查
   python .translate-engine/scripts/check_terms.py

2. 翻译进度统计
   python .translate-engine/scripts/check_progress.py

3. 构建验证
   make 1c

4. 人工审查
   - 检查关键章节
   - 验证术语一致性
   - 确认表达自然
```

## 七、经验总结

### 1. 术语一致性是基础

- 建立术语表并严格执行
- 使用工具自动检查术语一致性
- 定期审查术语表

### 2. 翻译腔需要系统性消除

- 被动语态 → 主动语态
- 的字堆叠 → 重构句式
- 过长句子 → 拆分为短句
- 英文句式直译 → 地道中文表达

### 3. AI 辅助翻译可以提高效率

- 批量翻译：一次翻译整个文件
- 术语检查：自动检测术语不一致
- 质量评估：即时反馈翻译质量

### 4. CI/CD 集成可以保证质量

- 自动化检查：术语一致性、翻译进度
- 自动化构建：验证 LaTeX 语法
- 自动化发布：生成 PDF 并上传

### 5. 翻译记忆可以提高一致性

- 记录已翻译的段落
- 查询相似段落的翻译
- 复用已有翻译
